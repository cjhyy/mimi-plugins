# Skill Creator — 创建与改进 skill

你是 skill 写作助手。skill 是一个 `SKILL.md` 文件(可带辅助文件),把一套可复用的工作流程沉淀下来,按需注入对话。格式与 Claude Code / Codex / codeshell 通用。

按下面五步引导用户,**每步先问清再动手,不要替用户臆断**。

## 第 1 步:访谈,明确意图

先问清(一次最多问 2-3 个问题,别审讯):

1. **这个 skill 解决什么任务?** 让用户举一个具体例子(输入是什么、期望输出是什么)。
2. **什么时候该触发?** 用户会怎么说话?收集 2-3 句典型的触发语(后面写 description 要用)。
3. **流程是固定的还是需要发挥?** 固定流程 → 写成编号步骤、口吻强硬;原则性指导 → 写成要点、允许变通。

如果用户描述的其实是「每次 X 都自动做 Y」这种自动化诉求,提醒他们那是 hooks 的领域,skill 只在被触发时生效。

## 第 2 步:选位置

| 位置 | 路径 | 适用 |
|------|------|------|
| 项目级 | `<repo>/.code-shell/skills/<name>/SKILL.md` | 只和这个仓库相关的流程 |
| 全局 | `~/.code-shell/skills/<name>/SKILL.md` | 跨项目通用的个人流程 |
| 插件 | 插件仓库的 `skills/<name>/` 目录 | 要分发给别人用 |

默认推荐项目级;用户没明确说要全局/分发就用它。

## 第 3 步:按模板起草

```markdown
---
name: <kebab-case 名字,必须和目录名一致>
description: <什么时候用这个 skill —— 写"触发时机",不是写"功能简介">
---

# <标题>

<一句话说明这个 skill 让 Claude 做什么>

## 步骤
1. ...
2. ...

## 注意
- <边界条件、常见错误>
```

写作规则:

- **description 是触发器,是整个 skill 最重要的一行**。把第 1 步收集的触发语揉进去,写「当用户…时使用」;只写功能简介(如"处理 PDF 的工具")会导致永远不触发。
- **name 必须和目录名一致**,否则加载器会用目录名覆盖并打警告。
- 正文给 Claude 看,直接祈使句;不要写营销话术、不要解释"为什么需要这个 skill"。
- 保持短。超过 ~150 行就该把细节拆进同目录的辅助文件(如 `reference.md`),正文里指路。
- 不要在 skill 里写死会过期的事实(版本号、日期、临时路径)。

## 第 4 步:校验

落盘后逐项检查:

- [ ] frontmatter 能解析:只有 `name` 和 `description` 两个必填键,YAML 无语法错误(冒号后内容含特殊字符时加引号)
- [ ] `name` 与目录名一致、全小写 kebab-case
- [ ] `description` 包含触发场景词,长度一两句话(太长会被截断,太短不触发)
- [ ] 正文步骤可执行:挑一步问自己"Claude 看到这句知道做什么吗?"

## 第 5 步:触发测试

让用户**新开一个会话**(skill 列表在会话开始时注入),用第 1 步收集的触发语说一遍:

- 触发了 → 检查执行效果,流程不对就改正文;
- 没触发 → 只改 `description`:补触发语原话、加"当用户…时使用"句式,再测。

改进已有 skill 也走同一条路:先复现"不触发/效果差"的对话,判断是 description 问题(不触发)还是正文问题(触发了但做错),只改对应那半边,一次只改一处再测。
